When should you switch?
Start with a monolith. Move to microservices when: team grows past ~10 engineers, different features need different scaling profiles, or deployment cycles are blocking each other. Netflix, Uber and Amazon all started as monoliths.

Core Components
Every building block in the architecture — what it does and why it exists
🔀
API Gateway
:5000 — YARP / Ocelot
Single entry point for all clients. Handles rate limiting, JWT validation, load balancing, SSL termination, and routing requests to the right microservice. Clients never talk directly to services.
YARPOcelotKong
🔐
Auth Service
:5001 — .NET Core + JWT
Issues and validates JWT tokens. Handles login, registration, refresh tokens, and OAuth flows. All other services validate JWT signatures independently — no calls back to auth service per request.
JWTOIDCRefresh Tokens
📦
Product Service
:5002 — .NET Core
Manages product catalog, inventory levels, search, and pricing. Uses MongoDB for flexible product schemas. Publishes events when inventory changes so other services can react.
MongoDBElasticsearch
🛒
Order Service
:5003 — .NET Core
Handles cart, order creation, order status tracking. Publishes OrderPlaced events to the message bus. Other services (Payment, Notification) subscribe and react. Uses Saga pattern for distributed transactions.
PostgreSQLSaga pattern
💳
Payment Service
:5004 — .NET Core
Subscribes to OrderPlaced events. Calls payment gateway (Stripe/Razorpay), publishes PaymentSucceeded or PaymentFailed events. Idempotent — same payment request always produces same result.
StripeIdempotency
🔔
Notification Service
:5005 — .NET Core
Subscribes to multiple events (OrderPlaced, PaymentSucceeded, ShipmentSent). Sends email, SMS, and push notifications. Completely decoupled — other services don't know it exists.
SendGridTwilio
📬
Message Bus
RabbitMQ / Azure SB
The async backbone. Services publish events without caring who listens. Subscribers process events independently. Guarantees at-least-once delivery. Decouples publishers from subscribers completely.
RabbitMQAzure SBMassTransit
🔍
Service Discovery
Consul / Kubernetes
Services register themselves on startup. Other services find them by name, not hardcoded IP. Health checks remove unhealthy instances automatically. In Kubernetes, this is built-in via DNS.
ConsulK8s DNS

Essential Microservice Patterns
Design patterns that solve the hardest distributed systems problems
🔄
Saga Pattern
Manages distributed transactions across services without a 2-phase commit. A series of local transactions, each publishing events. If one fails, compensating transactions roll back previous steps.
OrderPlaced → PaymentDebited → InventoryReserved
Payment fails → CompensateOrder → Refund issued
⚡
Circuit Breaker
Prevents cascading failures. When a service starts failing (e.g. 50% error rate), the circuit "opens" and immediately returns an error instead of waiting for timeout. After a cooldown, it half-opens to test recovery.
Closed (normal) → Open (failures) → Half-Open → Closed
// Polly library in .NET Core
📮
Outbox Pattern
Solves the dual-write problem: saving to DB and publishing an event in a single atomic operation. Write the event to an "outbox" table in the same DB transaction. A background worker publishes it to the bus.
BEGIN TRANSACTION
INSERT INTO Orders ✓
INSERT INTO Outbox ✓
COMMIT — then publish async
🎭
CQRS
Command Query Responsibility Segregation. Separate the write model (commands: CreateOrder) from the read model (queries: GetOrders). Write to PostgreSQL, read from a denormalized MongoDB or Redis read model optimized for queries.
Command: CreateOrderCommand → Write DB
Query: GetOrdersQuery → Read DB (fast)
// MediatR library in .NET
🔁
Retry + Backoff
Transient failures (network blips, temporary overload) are normal in distributed systems. Automatically retry failed calls with exponential backoff — wait 1s, 2s, 4s, 8s — to avoid thundering herd.
// Polly in .NET Core
Policy.Handle<HttpRequestException>()
.WaitAndRetryAsync(3,
i => TimeSpan.FromSeconds(Math.Pow(2, i)))
🌊
Bulkhead
Isolates failures like a ship's compartments. Limit thread pool / connection pool per downstream service. If Payment Service is slow, it only consumes its own resource pool — Order Service isn't affected.
PaymentClient: max 10 concurrent calls
ProductClient: max 20 concurrent calls
// Polly Bulkhead policy
Challenges & How to Solve Them
Microservices introduce distributed systems complexity. Here's what to expect and how to handle it.
🔍
Distributed Tracing
A single user request touches 5 services. When something fails, how do you find where? Traditional logs are useless — you need to correlate across services.
✓ Solution: OpenTelemetry + Jaeger / Zipkin. Propagate TraceId across service boundaries. One trace ID follows the request end-to-end.
💸
Distributed Transactions
You can't use SQL transactions across services with separate databases. "Charge payment AND place order" must be atomic — but they're in different DBs.
✓ Solution: Saga Pattern with compensating transactions. Accept eventual consistency. Use the Outbox Pattern for reliable event publishing.
📡
Service-to-Service Latency
A monolith makes in-process calls (nanoseconds). Microservices make network calls (milliseconds). 10 services in a request chain = 10× the latency.
✓ Solution: Async messaging where possible. gRPC for sync calls (faster than REST). Redis caching. Avoid chatty service-to-service calls — aggregate in a BFF.
🔀
Service Discovery
Services scale up and down. Their IPs change constantly. Hardcoding "http://payment-service:5004" breaks when IP changes or new instances spin up.
✓ Solution: Consul or Kubernetes DNS. Services register on startup and deregister on shutdown. Health checks remove dead instances automatically.
📋
Data Consistency
Services have their own DBs. A product shown in Order Service may be out of date vs Product Service. Eventual consistency means data is temporarily inconsistent.
✓ Solution: Accept eventual consistency. Use events to sync data. Design UIs to handle stale data. Use the Saga pattern for critical flows.
🔒
Auth Across Services
Each service needs to verify who the caller is. Calling auth service from every other service on every request creates a bottleneck and tight coupling.
✓ Solution: JWT tokens. Auth service issues signed tokens. Every other service validates the signature independently — no auth service calls per request.
🔄
Cascading Failures
Payment Service goes down. Order Service waits for it, consuming threads. Gateway waits for Order Service. Everything grinds to a halt from one failure.
✓ Solution: Circuit Breaker pattern (Polly library). Timeouts everywhere. Bulkhead isolation. Fail fast — don't let one slow service drag down others.
📦
Deployment Complexity
10 services = 10 CI/CD pipelines, 10 Docker images, 10 Kubernetes deployments. Infrastructure complexity grows massively vs a single monolith deployment.
✓ Solution: Docker + Kubernetes. Helm charts for templating. GitOps with ArgoCD. .NET Aspire for local development orchestration.
